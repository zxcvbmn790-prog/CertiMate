-- Git 기본 세팅 --
작업폴더에서 git 쓰고 싶으면 "git init"부터 입력 -> Git이라는 소프트웨어가 작업을 감시하기 시작함

-- Git 기록 방법 --
git add 파일명 => 파일 현재 상태를 기록하는 방법, 기록할 파일 고르기
git commit -m '메모' => 고른 파일 기록 명령
# 버전 생성이라고도 한다

-- Sturucture --
[작업폴더] -git add(스테이징한다)-> [staging area] -git commit-> [repository]

-- Staging Bonus --
git add 파일1 파일2 => (공백으로 구분)
git add . =>  현재 작업 폴더의 모든 파일 스테이징

-- 추가 명령어 --
git status => 상태창 열기 : 어떤 파일들이 스테이징 되어있는지 수정되었는지를 알려준다
git log --all --oneline : 커밋한 기록을 보기 + 커밋아이디도 볼 수 있다

-- 차이 비교 --
git diff => 최근 commit vs 현재 파일 차이점을 보여준다 (j, k로 스크롤. q로 나가기)
    단점 => 터미널의 한계로 보기 복잡, code가 복잡해지면 보기가 힘들다
git difftool => vim에디터로 더욱 시각적으로 좋게 차이점을 비교해준다 (hjkl이 방향키. :q혹은 :qa로 종료)
git difftool 커밋아이디 => 현재파일과 특정 커밋과의 차이점 비교 가능(두개까지 공백구분으로 비교 가능)

#3
-- Branch --
branch : commit의 복사본
git branch 브랜치명 => 브랜치 생성
git switch 브랜치명 => 브랜치로 전환
git log --oneline --all --graph => 그래프 형식으로 로그를 보여준다

-- Details --
HEAD : 나의 위치 ((HEAD -> main => 현재 main에 있다))

-- merge --
먼저 메인이 될 브랜치로 이동 후,
git merge 브랜치명 => main에 합칠 브랜치명을 명시하고 합친다
* 같은 파일의 같은 내용을 수정 시에는 어떤걸 남길지 물어본다
이후 다시 add와 commit을 하면 끝 



# 협업 방식
-- GitFlow 전략 --
1. main
2. develop
3. feature
4. release
5. hotfix
▼ -- ▼ -- ▼
main 브랜치(안전)
ㅣ
▼
develop (브랜치, 신기능 개발하는 곳)
ㅣ
▼
feature/기능명 (브랜치, 신기능 개발해보고 괜찮으면 develop에 merge)

합쳐진 브랜치들을 release라는 브랜치를 만들고 테스트를 해본다
완성이 되면 main에 다시 합친다 -> 배포
& release를 develop에도 merge한다 -> 계속 개발을 해야하기 때문

버그가 생기면 hotfix라는 브랜치를 바로 쳐서 버그 수정을 하고 main에 merge(develop에도)하고 배포한다

pros : 안정적으로 버전별 배포 가능
cons : CI/CD 이런거 하는 곳은 안좋아함

-- Trunk-based --
"브랜치 하나만 잘 관리하자"
main 브랜치 하나만 유지하되, 새로운 기능들을 추가할 시, feature1, 2 등으로 브랜치를 만들어서 바로 main에 merge

pros : 소스코드가 한 곳에만 있어서 편하다
cons : 테스트가 많이, 자주 필요하다
안정화된 프로젝트 들이 잘 쓴다
